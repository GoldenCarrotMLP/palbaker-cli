8. **Bugfix: CLI Success Masking on Subprocess Pipeline Failures**
   - **The Issue**: When pushing, decompiling, or extracting a mod, if the underlying Python subprocess (`build_mod.py`, `decompiler.py`, or `extractor.py`) threw a fatal error (e.g. material binding failures in Unreal Editor), the main CLI `palbaker_cli.py` still exited with a `0` (Success) status code. This masked failures, preventing the console from entering an error state and incorrectly showing a green "Success" toast.
   - **The Root Cause**: Subprocess execution blocks in `palbaker_cli.py` intercepted the `proc.returncode` and outputted a JSON result with `{"status": "error"}` but never propagated the non-zero exit status back to the shell process level. As a result, Tauri's `run_cli` parser was fooled into believing the execution was fully successful.
   - **The Fix**: Updated `palbaker_cli.py` to strictly propagate subprocess exit codes on failure. Added `sys.exit(proc.returncode)` (for build mod delegated actions) and `sys.exit(1)` (for extract/decompile pipeline failures) when their respective execution statuses indicate errors. This correctly surfaces exit codes to Tauri, forcing the console to adopt the error state and trigger red failure toasts on the frontend.
